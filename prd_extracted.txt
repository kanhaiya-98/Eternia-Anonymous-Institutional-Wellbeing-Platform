Executive Summary &amp; Product Vision Eternia is a controlled-anonymous institutional student wellbeing platform built for colleges, universities, and professional institutions. It provides a structured, secure, and anonymity-preserving environment where students can access professional counselling, peer support, emotional release tools, sound therapy, and self-help resources — all mediated through institutional control rather than open public registration. The platform operates as a closed, invite-only ecosystem. Access is gated through a three-layer institutional validation system: an Eternia Institution Code, a Grievance Officer QR scan, and device-level binding. No student is identifiable to other platform users. Personal identity data is encrypted, institution-controlled, and accessed only under defined crisis escalation protocols. This document serves as the complete technical specification for the developer agency responsible for building Eternia. It defines the product architecture, access flows, feature modules, infrastructure strategy, security model, compliance posture, and delivery milestones. Strategic Architecture Note — Version 3.0 This revision maintains the full product vision and all security principles of previous specifications. The only material change is to the infrastructure and deployment strategy. Previous versions specified enterprise-grade infrastructure (ECS clusters, autoscaling groups, Redis clusters, PgBouncer, multi-node HA setups) appropriate for a mature SaaS product at scale. Version 3.0 re-architects the deployment layer for an early-stage startup context: — Phase 1 targets 0–10,000 users with infrastructure costing Rs. 8,000–15,000/month. — The codebase is designed as a modular monolith: clean internal boundaries, stateless API, horizontal scaling possible without architectural rework. — The full scaling path to 1,00,000 users remains defined and achievable. — No product features, security controls, or compliance requirements have been removed. 1.1 Platform at a Glance Platform Type Controlled-anonymous, institutional SaaS — not an open public app Primary Users Students of partnered institutions including Schools, Colleges, and Universities accessed exclusively via institutional tie-ups . Core Identity &amp; Platform Model Eternia is an anonymous institutional wellbeing platform. The word 'anonymous' here carries a precise technical meaning: students do not reveal personal identity publicly or to peers at any point within the platform. Their username — self-chosen during registration — is the only identity visible in any peer-facing context. Critically, Eternia is not a social media application, not a public mental health app, and not a chatbot. It is a structured, moderated, institution-controlled environment. Every access point, every feature, and every data operation is scoped to the institutional relationship. What Eternia IS A closed institutional ecosystem A controlled anonymous platform A structured wellbeing service Institution-governed data environment A professional-grade SaaS product DPDP Act-compliant data processor A scalable, upgradeable platform What Eternia IS NOT An open public application A social media or chat platform A diagnostic or clinical tool A Firebase hobby project A serverless architecture A consumer wellness app A one-time MVP with no scale path 2.1 Institutional Tie-Up Model Eternia reaches students exclusively through institutional partnerships. An institution (college, university, professional body) signs an agreement with Eternia, receives an Eternia Code for their cohort, designates a SPOC (Single Point of Contact) / Grievance Officer, and receives a prebuilt QR-coded ID. The SPOC controls all student onboarding within their institution. No student can self-register independently. This model creates a commercially sustainable B2B2C structure: Eternia sells institutional packages (with Care Credit bundles), institutions distribute access to students, and students consume services using credits. The SPOC dashboard gives institutions full visibility over anonymous aggregate wellbeing metrics, session utilisation, and care credit consumption without ever exposing individual student identities. 3. Access &amp; Onboarding Architecture The five-step onboarding architecture is the defining differentiator of Eternia. It simultaneously enforces institutional authenticity, prevents public misuse, creates device-level accountability for crisis traceability, and establishes the anonymity guarantee for normal platform use. This architecture must be implemented exactly as specified and is not subject to simplification. 3.1 Step-by-Step Onboarding Flow Step 1 — Eternia Code Entry Upon opening the mobile application, the user is presented with a single-field entry screen requesting the Eternia Institutional Code. This code is provided to institutions at the time of their partnership agreement with Eternia. Purpose Restrict access exclusively to partnered institutions; prevent public misuse Implementation POST /auth/validate-code — backend validates HMAC-hashed code against institutions table On Success Returns institution metadata and enables Step 2; no user record created yet On Failure Generic error — no institution metadata revealed to prevent enumeration attacks Security Note Code is stored as a one-way HMAC hash — plaintext never stored server-side Step 2 — Grievance Officer ID Scan After Eternia Code validation, the app launches the device camera and requests that the student scan the SPOC / Grievance Officer's prebuilt QR code. This QR code is provided to the institution as part of their onboarding package. Purpose Institution-level controlled onboarding; no random sign-ups from the internet QR Code Content HMAC-SHA256 signed payload containing: institution_id, spoc_id, issuance timestamp, TTL TTL Strategy QR code payload has a 24-hour validity window; SPOC regenerates via dashboard as needed Implementation POST /auth/validate-spoc-qr — validates HMAC signature and institution match On Success Issues a short-lived onboarding session token (TTL: 15 minutes) to proceed to Step 3 Replay Attack HMAC signature + timestamp validation prevents code replay. Nonce stored in Redis. Step 3 — Device Primary Access Binding Upon successful QR scan, the application generates a device fingerprint derived from hardware and OS identifiers and registers it as the Primary Access Device for this account. Purpose Prevent account sharing; maintain crisis traceability if escalation is required Fingerprint Composite of: device UUID, OS version hash, manufacturer hash — AES-256 encrypted before storage Implementation Encrypted device_id stored in users table; validated on every subsequent authenticated request Anonymity Note Device binding is for internal traceability only — not exposed to any peer or public-facing UI Reset Protocol Only SPOC can reset device binding via institution dashboard (e.g., device lost/replaced) Step 4 — Credential Setup The student selects a username and sets a password. No real name, email, or phone number is required or collected at this stage. The username is unique within the institution scope. Username Unique within institution; visible to intern/expert in sessions — never real name Password Minimum 8 characters; stored as bcrypt hash (cost factor: 12) Implementation POST /auth/register — validates onboarding token from Step 3, creates user record No PII No email, phone, or real name collected in this step by design Step 5 — Private Profile Completion After credential setup, the student is prompted to complete their private profile. This data is encrypted and never surfaced in any peer-facing or public context. It is used exclusively under defined crisis/escalation protocols. Data Collected • Emergency Contact Details (name + phone) with ownership declaration. • Student Verification Identifier. The student must specify whether the emergency contact number belongs to themselves or to another individual. If marked as another individual, an additional mandatory field captures the relationship to the contact person. Verification Identifier varies by institution type: • University / College Model: Student submits APAAR / ABC ID for institutional validation. • School Model: Student submits ERP ID (institution-issued student admission number or serial ID). The verification identifier depends on the type of partnered institution. Eternia supports two verification models: Model A — University / College Verification (APAAR / ABC ID) For higher education institutions, students verify their academic identity using their APAAR / ABC ID. Verification Flow : Student enters APAAR / ABC ID during Step 5. Backend sends the ID to the institutional verification endpoint. Institution database checks if the APAAR ID exists in their records. Response returned: { "YES" } or { "NO" }. If YES: users.apaar_verified = true Encrypted APAAR ID stored in user_private.apaar_id_encrypted Raw APAAR value immediately discarded from memory If NO : Generic validation error returned. Model B — School Verification (ERP ID) For school institutions, verification is performed using the ERP ID assigned to every student during admission. ERP ID is typically a unique serial number or admission ID issued by the school management system. Verification Flow : Student enters ERP ID during Step 5. Backend sends ERP ID to the institution ERP verification endpoint. School database verifies if ERP ID exists. Response returned: { "YES" } or { "NO" }. If YES: users.erp_verified = true Encrypted ERP ID stored in user_private.erp_id_encrypted Raw ERP ID immediately discarded from memory If NO : Generic validation error returned. Security Handling Both APAAR ID and ERP ID are treated as Sensitive Personal Data under DPDP Act 2023. Security controls include: • AES-256-GCM encryption before database write • Raw identifiers never stored in plaintext • Identifiers never returned through any API response • Access permitted only through authorized escalation protocols 3. 4 Emergency Contact Escalation Consent During private profile completion, the student must acknowledge an escalation consent checkbox before submitting emergency contact information. Consent Statement (Displayed in App) “If the platform detects a high-risk situation requiring intervention, the system may notify your institution’s designated SPOC. In such cases, your username and the emergency contact number you provided may be shared with the SPOC to enable immediate support action.” Purpose • Allows the institution to respond quickly during crisis situations • Enables SPOC to contact the emergency number if necessary • Maintains transparency with the student regarding escalation procedures Technical Handling When escalation protocol triggers: The system sends the following key-value pair to the SPOC escalation dashboard: username : &lt;student_username&gt; emergency_contact : &lt;encrypted_phone&gt; No additional personal identity fields (APAAR, ERP, device fingerprint) are shared in this notification. Full identity reveal still requires the formal escalation approval workflow defined in Section 14.2. 4. Home Page — Core Feature Portals Upon completing onboarding, the student lands on the Eternia Home Page, which surfaces five primary portals and a set of supplementary self-help tools. Each portal is architecturally designed as an independent module within the monolith — with its own service layer, route group, and data boundary. This internal separation ensures that each module can be extracted as a standalone service in a future microservices migration without codebase refactoring. 4.1 Expert Appointment Portal The Expert Appointment Portal enables students to book time-bound, structured sessions with verified mental health professionals — institutional counsellors or Eternia-partnered psychologists. Booking is mediated through the Care Credits system. Expert availability is defined via a slot management system in the SPOC/Admin dashboard Students browse available slots, select preferred time, and initiate a booking System checks credit balance, reserves the slot (15-minute hold), and deducts credits on confirmation Booking confirmation triggers push notifications to both student and expert Session conducted via in-app video (Daily.co SDK or equivalent) or an institutional external link Post-session status update triggers session logging and feedback prompt 4.2 Peer Connect — Intern-Moderated Anonymous Chat Peer Connect enables students to engage in real-time anonymous conversations with verified, trained psychology interns under institutional supervision. It is not open social chat — it is a structured, moderated peer-support interaction designed to provide early-stage emotional support while preserving the anonymity of the student. Student enters the Peer Connect portal and is presented with a list of available interns currently active on the platform, including limited non-identifying attributes such as training badge, focus areas, and availability status Student selects a preferred intern and initiates a session request; the selected intern receives a push notification and must accept the request within a time window, after which the request expires and the student may select another intern . Real-time communication (audio call or chat) operates via Socket.IO rooms — the room identifier is the session UUID, ensuring that no personal identity data is exposed within the communication channel Interns participating in Peer Connect must complete the platform’s mandatory training program before gaining access to live sessions; intern access to Peer Connect remains locked until training completion and administrative approval . Interns can flag a session for escalation at any time without breaking student anonymity; flagged sessions may trigger review by a supervising expert or institutional authority Session transcripts are processed through the platform’s monitoring and compliance layer; storage occurs only under defined escalation conditions and remains encrypted with restricted access controls . Care Credits (ECC) are deducted per completed session in accordance with the platform’s credit model, and session metadata (session ID, timestamps, intern ID reference, escalation flags) is logged to the peer_sessions table for audit and analytics purposes 4.3 BlackBox — Controlled Anonymous Expression The BlackBox is Eternia's most sensitive and institutionally governed module. It is not social media, not a journaling repository, and not a persistent voice storage system. It provides a controlled anonymous interaction environment designed for emotional expression and guided support while preserving strict identity separation under normal platform use. Students access BlackBox through an anonymized session interface represented as a neutral interaction environment. Neither the student nor the responding professional is exposed to personal identity information during standard operation. The module prioritizes psychological safety through anonymity while maintaining institutional accountability through defined escalation safeguards. In-App Economy — Eternia Care Credits (ECC) The Eternia Care Credit (ECC) system is the internal economic engine of the platform. Credits are the sole mechanism through which students access billable services. The Care Credit ledger is designed to the standards of an immutable financial transaction log — no direct balance mutation is ever permitted. 5.1 Credit Architecture Principles Credit balance is never stored as a mutable field — it is always computed as SUM of credit_transactions.delta for a given user_id Every credit operation (earn, spend, grant, purchase) creates an immutable record in credit_transactions This ensures a full audit trail of all ECC movements and prevents accidental or malicious balance manipulation Negative balance prevention enforced at API layer before any deduction is committed 5.2 Credit Lifecycle Credit Source Description Institution Grant SPOC bulk-allocates credits to all enrolled students via institution dashboard; typically at term start In-App Purchase Student purchases credit top-up via Razorpay (India) integration; credits added post-payment confirmation Engagement Earn Platform-defined earn events (Quest Card completion, session feedback) award credits; institution-configurable Credit Spend Deducted when booking Expert sessions, initiating Peer Connect sessions, accessing premium BlackBox features Package Allocation Institutional SaaS packages include pre-allocated credit bundles tied to plan tier 5.3 ECC Monetisation Layer The Care Credit system provides Eternia with multiple monetisation vectors: institutional package fees (including bundled credits), in-app credit top-up revenue (2% Razorpay fee absorbed or passed to user), and premium content unlock revenue. Credit expiry logic is configurable per institution and implemented via scheduled BullMQ job in Phase 2. 6. Modular Monolith Architecture Strategy The foundational architectural decision in Version 3.0 is to build Eternia as a Modular Monolith for the Phase 1 launch. This is not a compromise or a technical shortcut — it is the correct architecture for an early-stage SaaS product targeting 0–10,000 users with a small engineering team and a defined scaling roadmap. 7.1 What is a Modular Monolith? A modular monolith is a single deployable application unit (one Node.js process, one Docker container) that is internally structured into clearly separated, loosely coupled modules — each with its own domain logic, service layer, route group, and database schema namespace. It is the opposite of spaghetti code; it is disciplined internal architecture that happens to share a deployment boundary. The critical property of a well-designed modular monolith is that each module can be extracted into a standalone microservice in the future without requiring changes to the remaining modules or the database schema. The extraction is a deployment decision, not a refactoring exercise. 7.2 Eternia Module Boundaries Module Owns Does Not Touch Auth / Onboarding users, user_private, institutions, device sessions Business logic of other modules Expert Appointments appointments, expert_availability, session_notes Credit ledger logic (calls Credits module) Peer Connect peer_sessions, peer_messages (via Socket.IO rooms) BlackBox entries, appointment data BlackBox blackbox_entries, escalation_flags Session data, appointment data Sound Therapy sound_content, play_history (aggregate only) User PII, session data Self-Help Tools quest_cards, tool_interactions Credits (earn events passed to Credits module) Care Credits (ECC) credit_transactions, credit_balance (computed view) No direct write to other modules SPOC / Admin institution_config, audit_logs, escalation_requests Direct read/write to other module tables 7.3 Scaling Guarantee The modular monolith design provides a firm scaling guarantee to investors and institutional partners: when Eternia grows to the point where a specific module (most likely Peer Connect or BlackBox, due to high write volume at scale) requires independent scaling, it can be extracted as a standalone microservice. The remaining monolith continues operating unchanged. No feature regression. No re-architecture. No downtime migration. Architecture Invariants — All Phases The following properties are maintained across all architecture phases and must be enforced by the developer team: 1. Stateless API Servers — All session state lives in Redis or the database, never in application memory. This means any number of API server instances can serve any request at any time. 2. Idempotent Operations — All write endpoints (especially credit deductions, session creation) are idempotent with client-provided request IDs to prevent duplicate operations. 3. Async by Default — All non-latency-critical operations (push notifications, email, escalation alerts, credit expiry) are handled via BullMQ queue, never synchronously in the request path. 4. Database Indexes — institution_id, user_id, status, created_at indexed on every core table from Day 1. This prevents query degradation as row count grows. 5. Cursor-Based Pagination — All list endpoints use cursor pagination, never OFFSET/LIMIT. OFFSET-based pagination degrades catastrophically at scale. 8. Recovery Setup – Account Continuity &amp; Identity Recovery Layer The Recovery Setup provides a secondary, non-PII account continuity mechanism designed to enable secure account recovery while preserving Eternia’s anonymity guarantees. Recovery configuration is optional during onboarding and may be enabled by the student at any time through the profile settings interface. The recovery layer does not rely on phone numbers, email addresses, or external authentication providers. Instead, identity verification operates through user-defined memory-based credentials stored in encrypted form. • Recovery Setup is enabled voluntarily by the student through profile settings after successful onboarding. • The recovery mechanism consists of two independent authentication layers: — Fragment Word Pairs: the student defines up to three keyword-based fragment pairs known only to the account holder. — Emoji Pattern Selection: the student selects a predefined emoji pattern combination from a controlled grid-based selection interface. • Fragment pairs and emoji pattern selections are encrypted using AES-256-GCM prior to database write and stored within a dedicated recovery_credentials table with restricted access permissions. • Recovery credentials are write-only from the application perspective and are never returned in any API response. Recovery Scenarios Recovery operates under the following defined cases: Case 1 — Same Device, Password Forgotten • Student enters username and initiates account recovery. • Backend validates fragment word pairs and emoji pattern against encrypted recovery credentials. • On successful verification, password reset access is granted without identity disclosure. Case 2 — New Device, Password Forgotten • Student initiates recovery using fragment pairs and emoji pattern. • Upon successful validation, a new Eternia ID and device binding are issued. • Historical activity, credits, and account state are securely migrated to the newly issued identity while maintaining audit continuity. Case 3 — Complete Credential Loss (Username and Recovery Credentials Forgotten) • Student initiates fresh onboarding and receives a new Eternia ID. • Previous account access cannot be restored. • Prior account progress and activity remain permanently inaccessible and may be scheduled for deletion in accordance with data retention policy. Security Controls • Recovery attempts are rate-limited and logged within audit_logs to prevent brute-force attempts. • Recovery verification requires successful validation of both fragment credentials and emoji pattern layers. • SPOC and institutional roles have no visibility into recovery credentials. • Eternia Admin access is restricted to operational audit visibility only and does not permit recovery credential disclosure. 1 8 . Escalation Model &amp; Live Replacement Architecture The Escalation Model defines the protocol through which a session is transferred from an intern, therapist, or expert to a higher-qualified mental health professional when the system detects that the situation exceeds the handling capacity of the current session moderator. This mechanism ensures timely intervention while maintaining the student’s anonymity and preserving institutional oversight. Escalation occurs when a session crosses AI risk threshold Level 3 or manual escalation is triggered by intern/expert/therapist. Escalation Triggers • AI high-risk detection • Intern escalation flag • Therapist escalation flag • Expert escalation request The AI system classifies session risk levels using the following structure: Level 0 — Normal conversation Level 1 — Emotional distress monitoring Level 2 — Support alert to institution (no identity exposure) Level 3 — Critical escalation required Only Level 3 triggers the live escalation mechanism. Live session replacement is implemented through a host-transfer mechanism within the video or audio session infrastructure. The following session frameworks may be used for implementation: • Daily.co Video SDK • Twilio Programmable Video • Zoom SDK Upon expert acceptance, the system performs the following actions: • Current session moderator is removed from the session room • M.Phil psychologist joins as the primary session host • Session control privileges are transferred to the new host • The session room ID remains unchanged during escalation to preserve connection continuity for the student. Only moderator roles are updated. • Session metadata is updated in the database All escalation events are recorded for institutional compliance and legal audit purposes. When escalation occurs, the system stores a limited encrypted session record containing : • Escalation trigger timestamp • 10 seconds of session data preceding the escalation trigger • 10 seconds of session data following the escalation trigger • Escalation trigger source (AI or manual) • Session ID and participating role identifiers Full session recordings are not permanently stored unless required under formal escalation procedures. All stored escalation data remains encrypted and accessible only to Eternia Admin under authorised review conditions. In cases where escalation reaches Level 3 severity, the system may notify the institution’s designated SPOC through the escalation notification system. The SPOC notification contains: • Escalation alert indicating that a student within the institution may require immediate support • Session reference ID • Escalation timestamp The notification does not reveal the student’s identity or session content unless a formal identity reveal request is initiated through the established escalation approval protocol 19.1 Selective Transcription Monitoring All real-time sessions pass through the AI selective transcription layer. Eternia Care Credit ( ECC ) The Eternia Care Credit (ECC) system operates as a controlled in-app currency used to regulate access to interactive wellbeing services within the platform. Credits are distributed to students through institutional allocations and platform engagement mechanisms and are consumed when accessing support services. ECC is not a monetary instrument and does not represent financial value outside the platform. It functions strictly as an internal usage credit system designed to manage fair access to platform resources. ECC Credit Allocation Each student account receives a limited ECC allocation based on the institution’s subscription plan. Credits may be distributed either: • Monthly allocation • Annual allocation (institution-configurable) ECC credits are stored within the platform credit ledger and tracked through the immutable credit_transactions system . ECC Credit Spending Structure ECC credits are consumed when students access support services that require moderation, professional time, or system resources. Credits may be spent in the following modules: Module Description BlackBox Anonymous support interaction with therapist or moderation system Peer Connect Real-time session with trained psychology intern Expert Appointment Scheduled session with certified mental health expert Each session deducts ECC according to configured pricing rules. All credit deductions are recorded as immutable ledger entries within credit_transactions ECC Credit Earning Activities Students may earn limited ECC credits through engagement with non-clinical wellbeing activities designed to promote emotional self-regulation and platform engagement. Eligible earning activities include: • Sound Therapy • Wreck the Buddy • Tibetan Bowl interaction • Quest Card completion These activities trigger credit earning events through the engagement system. Daily ECC Earning Cap To prevent credit farming or misuse, the platform enforces a daily earning cap. Maximum ECC earnable per user per day: 5 ECC Any additional engagement activity beyond the cap will not generate additional credits until the next daily reset cycle (UTC midnight). ECC Low Balance Prompt When a student's wallet balance falls below a defined threshold, the platform provides a gentle notification encouraging credit replenishment. Trigger Condition: wallet_balance &lt; 5 ECC User Prompt: “Your care energy is low. Refill gently.” Students may replenish credits through institutional grants, engagement earnings, or ECC bundle purchases where enabled. ECC Bundle Packs ECC bundle packs allowing students to purchase additional credits within the application. Purchases are processed through the platform’s payment gateway (Razorpay integration). Bundle purchase events create immutable credit grant entries within credit_transactions. ECC Stability Pool The ECC Stability Pool is a platform-level safety mechanism designed to ensure that students who temporarily reach zero credits can still access essential support services. Each student account contributes a small automated credit allocation into the shared pool. Contribution rule: 1 ECC automatically contributed per student per month The pooled credits are used exclusively for the following cases: • Funding Peer Connect sessions for students whose ECC balance has reached zero • Covering emergency Expert Appointment sessions when a crisis-level event occurs • Ensuring that no student is completely locked out of wellbeing support due to credit depletion Pool usage is controlled by internal system rules and cannot be manually accessed by users. All pool allocations are recorded as system-issued credit transactions for audit transparency. ECC Governance Rules The following governance constraints apply to the ECC system: • ECC cannot be transferred between students • ECC cannot be withdrawn or converted to currency • ECC balance is derived exclusively from immutable ledger records • ECC Stability Pool usage is system-controlled and logged • Institutions may configure allocation limits but cannot override ledger integrity ETERNIA — Technical Specification v3.0 | Modular Monolith Edition | Confidential Enterprise quality.
